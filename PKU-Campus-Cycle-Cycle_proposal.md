# 项目名称：燕园易骑：校园自行车循环回收平台 (PKU Cycle-Recycle Hub)

## 1. 目标问题 (Problem Statement)

北京大学校园内存在大量废旧自行车，这些自行车由于长期闲置而生锈，被学校统一堆放至空地，造成了严重的资源浪费与校园空间占用。 这一现象反映了校内自行车使用的深层困境：

- **毕业生处置难：** 许多学生在毕业离校时，由于搬运成本高或缺乏正规转让渠道，不知道如何处置自己的自行车，导致大量自行车被遗弃在宿舍区或教学楼下，最终沦为僵尸车。
    
- **新生购车负担重：** 新生在入学时通常面临较大的开支，购买新自行车价格不菲，而现有的二手交易往往散落在各种社交群聊中，信息碎片化且缺乏信誉保障。
    
- **环境与管理压力：** 大量废旧金属废弃物对校园环境造成污染，学校管理部门处理这些废旧自行车的行政成本也很高。
    

**项目愿景：** 本项目计划在毕业季从毕业生手中回收二手自行车，并在开学季提供给新生低价或免费购买。通过将校园内的二手自行车进行循环利用，不仅减轻了毕业生的离校负担，降低了新生的生活成本，更能减少资源浪费与环境污染，保护绿色校园，形成多方共赢的闭环生态。

**数据支持：** 前期 78 份问卷调查显示，超 76% 的新生愿意通过回收项目获取自行车；89% 的毕业生愿意参与回收，其中 65% 的毕业生愿意免费提供，这为项目的资金与货源可行性提供了坚实基础。

## 2. 目标用户 (Target Users)

- **核心供给方：** 即将毕业离校的学生。
    
- **核心需求方：** 刚入学的新生及校内其他有代步需求的师生。
    
- **管理运营方：** 负责线下车辆存放、成色鉴定及核销的学生志愿者/管理员。
    

## 3. 核心功能 (Core Features)

### 3.1 门户与宣传模块

- **首页入口：** 提供“捐赠/出售”与“寻找座驾”两个核心入口。
    
- **项目愿景展示：** 实时展示回收数据（如：已累计减少多少千克金属浪费）。
    

### 3.2 车辆回收模块 (Supply Side)

- **登记系统：** 毕业生上传照片、填写品牌、新旧程度（评分制）及意向价格（可勾选“免费捐赠”）。
    
- **线下预约：** 毕业生选择管理员设定的空闲时间段，前往指定地点交车。
    
- **入库核验：** 管理员核对实车后，在后台将该车状态改为“已入库，可匹配”。
    

### 3.3 智能匹配与抢先预约 (Demand Side)

- **需求画像：** 用户输入期望的价位、成色范围。
    
- **多项选择与锁定逻辑：**
    
    - 系统根据需求筛选出**多辆**符合条件的自行车供用户对比。
        
    - **实时库存校验：** 用户在浏览过程中，若某车辆已被他人预选或线下成交，系统将实时标记为“交易中”或“已售出”。
        
    - **预约匹配：** 用户选择心仪车辆后点击“预约线下试骑”，系统生成匹配凭证。
        
    - **库存冲突处理：** 若在用户下单前的短时间差内车辆已被售出，系统将弹出友好提示“抱歉，该车刚刚被其他同学抢先锁定，请查看其他推荐”，并引导用户重新选择。
        

### 3.4 社区反馈模块

- **建议广场：** 用户可对回收流程、匹配效率提出建议。
    
- **评价系统：** 成功领车/买车的同学可以留言分享，建立社区信任感。
    

### 3.5 个人中心与管理后台

- **个人足迹：** 查看已发布的车辆状态或已预约的匹配进度。查看历史交易记录。
    
- **管理员工作台：** 处理入库申请、更新车辆状态（下架/成交）、管理预约时间段。查看所有历史交易记录。
    

## 4. 技术方案 (Technical Approach)

- **前端方案：** 使用 HTML5 + Tailwind CSS 搭建响应式页面，确保在手机端浏览的流畅性。
    
- **后端支撑：** 选用 Firebase (Firestore) 实时数据库。利用其实时监听特性，解决多用户同时选车时的库存状态同步问题。
    
- **核心逻辑：** 1.  **原子化更新：** 使用 Firestore 的 Transaction 功能，确保在多人抢购同一辆车时，只有一个用户能预约成功。 2.  **筛选算法：** 在前端实现多条件过滤器，实时渲染符合要求的车辆列表。
